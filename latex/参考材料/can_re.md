## 5. 结果分析

### 5.1 Count+Entropy（CACR）是最稳妥的效率—效果折中

`count_entropy` 在**不改模型参数**的前提下，只替换候选池构建方式，就取得了：

- 更低的候选检索延迟：`3.472701ms -> 2.927849ms`
- 更高的吞吐：`287.96 -> 341.55 QPS`
- 更低的高代价索引调用率：`100% -> 68.25%`
- 最终推荐性能几乎不变：
  - `HR@10: 0.534080 -> 0.535948`
  - `NDCG@10: 0.384886 -> 0.385060`

这说明：**CACR 可以直接作为 nofix 的候选检索替换方案，而无需重新训练，就能带来实际推理收益。**

### 5.2 Entropy-Only 也很强，但不如 Count+Entropy 高效

`entropy_only` 的最终推荐指标略优于 `count_entropy`：

- `HR@10 = 0.537815`
- `NDCG@10 = 0.387397`

但其候选检索延迟更高：

- `3.132252ms > 2.927849ms`

因此如果目标是数据库式“**在维持效果下优先降检索代价**”，`count_entropy` 更均衡。

### 5.3 Cheap-Only 和 Count-Only 过于激进

这两种方案虽然延迟最低，但最终排序指标下降更明显：

- `cheap_only`: `NDCG@10` 降到 `0.366837`
- `count_only`: `NDCG@10` 降到 `0.369847`

说明完全或近乎完全跳过高代价候选源，会伤害最终推荐质量。

### 5.4 Random-Skip 低于自适应策略

`random_skip` 与 `count_entropy` 拥有接近的高代价索引调用率：

- `count_entropy`: `0.682540`
- `random_skip`: `0.698413`

但最终性能更差：

- `HR@10`: `0.531279 < 0.535948`
- `NDCG@10`: `0.377313 < 0.385060`

这进一步说明收益来自**自适应查询计划**，而不是简单地减少索引调用次数。

## 6. 最终结论

在 NYC 数据集、现有 nofix 3 epoch checkpoint 上，**只修改候选池构建方式，不重新训练模型**，CACR 依然有效：

1. **候选检索效率提升明显**：平均延迟下降约 `15.69%`，QPS 提升约 `18.61%`；
2. **最终推荐性能基本不受损，甚至略有提升**：`count_entropy` 的 `HR@10` 和 `NDCG@10` 均与 `full` 持平或略优；
3. **高代价索引调用率显著下降**：从 `100%` 降到约 `68%`；
4. **优于随机跳过**：证明 CACR 的收益来自中间结果驱动的自适应扩展，而不是随机削减检索。

因此，当前结果支持将 CACR 定位为：

> **面向 nofix 下一 POI 推荐的自适应多索引候选查询优化方法。**

在不改动主模型、不重新训练的条件下，它已经展示出较好的工程可替换性和数据库式效率优势。

## 8. 补充观察：为什么 128 候选反而略好于 256 候选

从当前结果看，**128 候选版 full** 相比 **历史原始 full（平均约 250 候选）**，最终指标确实略有提升：

- 历史原始 full：
  - `HR@10 = 0.525677`
  - `NDCG@10 = 0.382764`
  - `candidate_avg_size = 250.458450`
- 128 候选版 full：
  - `HR@10 = 0.534080`
  - `NDCG@10 = 0.384886`
  - `candidate_avg_size = 127.669468`

因此在这次实验中：

- `HR@10` 提升约 `+0.008403`
- `NDCG@10` 提升约 `+0.002122`

### 8.1 可能原因

这类现象并不罕见，核心原因通常是：**候选更多，并不等于最终排序更好。**

可能的解释包括：

1. **更大的候选池引入了更多噪声候选**
   - 候选池从约 128 扩大到约 250 后，新增候选里可能包含大量低质量项；
   - 这些候选虽然提高了覆盖率，但也会干扰最终 top-10 排序。

2. **精排模型在较小、较干净的候选空间中更稳定**
   - 当候选集合更紧凑时，高相关候选更集中；
   - 模型更容易把目标 POI 排在前列。

3. **候选压缩带来了某种“去噪”效果**
   - 对于 nofix 这样的 two-stage 结构，候选召回不仅影响 coverage，也影响精排难度；
   - 适度压缩候选空间可能减少长尾噪声，使最终 NDCG@10 略有提升。

4. **提升幅度较小，仍需谨慎解释**
   - 当前提升不是数量级变化，而是一次 test 上的小幅提升；
   - 因此更稳妥的说法应是：**128 候选至少不差于 256，且本次实验中略优。**

### 8.2 建议表述

建议不要直接写成“128 候选一定优于 256 候选”，而应写成：

> 在当前 nofix 3 epoch checkpoint 上，将候选池规模从约 256 收缩到 128，并未损害最终推荐性能，反而在本次测试中带来了小幅的 HR@10 与 NDCG@10 提升。这说明更大的候选池不一定带来更优排序结果，适度压缩候选空间可能减少噪声候选干扰。

### 8.3 与 CACR 的关系

这一观察也强化了 CACR 的合理性：

- CACR 的目标不是一味扩大候选池；
- 而是在**维持足够覆盖率**的前提下，尽量减少不必要的候选扩展；
- 如果额外扩展带来的主要是噪声而不是有效目标候选，那么提前终止检索反而可能有利于最终排序。

因此，从当前结果看，**“候选更多 ≠ 最终效果更好”** 这一点已经得到了经验支持，也为 CACR 的技术动机提供了额外实证依据。
